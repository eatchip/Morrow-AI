# Playbook: Spec 评审

> 用于**自审**（Spec 作者交付前）和**他审**（Reviewer 批准前）。
> 未通过清单的 Spec 不得进入 Plan 阶段。

---

## Reviewer 视角：每一条都要能从 Spec 中找到答案

### A. 问题定义清晰
- [ ] 读完 Context 能用一句话复述要解决什么问题
- [ ] 约束（业务 / 技术 / 时间 / 合规）显式列出
- [ ] 非目标（明确不做什么）显式列出

### B. 方案可评估
- [ ] 至少考虑了 2 个备选方案及其 trade-off
- [ ] 采纳方案的依据可被挑战（不是"我觉得"）
- [ ] 涉及成熟工业问题 → 引用了外部参考（走过 `research-method.md`）
- [ ] 若含 UI：方案能追溯到 [ADR 0005](../decisions/0005-design-north-stars.md) North Star 的某个维度（Linear 骨架 / Arc 血肉），偏离必须有显式理由

### C. 影响面完整
- [ ] 受影响文件清单完整（路径 + 操作类型）
- [ ] 涉及外部契约变更 → 显式列出兼容策略
- [ ] 涉及新依赖 → 列出许可证 / 维护状态 / 替代品

### D. 风险识别
- [ ] 状态所有权明确
- [ ] 不变量写出 1–3 条
- [ ] 异步 / 并发 / 生命周期风险已讨论
- [ ] 安全 / 合规触点已检视
- [ ] 性能关注点（如有）已量化
- [ ] 若含 UI：四态 + 极端数据态 + 响应式尺寸均已描述（见 [`DESIGN.md § 6`](../design/DESIGN.md#6-state-coverage-checklist)）

### E. 验证可执行
- [ ] 验证方式具体（哪层测试 / 哪些场景 / 验收标准）
- [ ] 对高风险假设有 Feasibility Check 或原型证据
- [ ] 回滚策略清晰（如何退、退到什么状态）
- [ ] 若含 UI：
  - 触发档位判定正确（🔴/🟡/🟢/⚪，不落两档之间）
  - 视觉稿载体与路径已指定；Figma 降级到 HTML 原型的情形有原因
  - 用户视觉确认状态已勾选（🔴/🟡 档位必须"已确认"后方可进入 Plan）
  - Token 使用声明齐全，无硬编码视觉属性的打算

### F. 可切分
- [ ] 能被拆为 ≥ 2 个独立可验证的 task
- [ ] 每个 task 完成后系统状态自洽（可中断）

---

## 常见退回理由（Reviewer 应直言不讳）

| 症状 | 退回理由 |
|---|---|
| Context 里只说"我觉得需要" | 缺乏问题驱动 |
| 方案直接贴结果，无 Alternatives | 跳过了对比判断 |
| 影响文件清单空 / 只写"若干" | 范围不清，无法评审 |
| 不变量空，"会写测试的"兜底 | 设计未经思考，测试会打补丁 |
| 验证一句"会跑测试" | 无具体验收 |
| Plan 可能性仅有一个巨型 task | 不可切分，风险不可控 |

## 作者自审清单（交付前再过一遍）

读自己的 Spec，问自己：

1. 如果我 **3 个月后** 重新读这份 Spec，能否独立把它实现？
2. 一个完全不认识这个模块的 reviewer 能看懂吗？
3. 我最不确定的那个假设，有没有验证过？
4. 如果执行过半发现方案错了，能否回滚？
5. 是否有任何我**知道但没写**的前提？把它补进 Context。

自审通过 → 提交给 reviewer。
