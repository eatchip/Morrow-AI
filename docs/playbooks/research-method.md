# Playbook: Research → Synthesize → Adapt → Verify

> 处理**成熟工业问题**的默认方法论。
> 目标：不重新发明轮子，也不盲目抄袭。

---

## 何时用

- 引入新技术 / 新架构模式
- 解决领域内"已经有人做过"的问题
- 评估竞品实现方式
- 写 Spec 前的"Research"环节

## 何时不用

- 纯本项目内部约定类问题
- 非常新的、无成熟参考的问题（用 Feasibility Check 而非本 playbook）

---

## Phase 1 — Research

目标：**找到 2–3 个值得学习的参考**，不是穷举。

步骤：
1. 定义问题的**领域标签**（e.g.「scheduled agent task queue」）
2. 在以下来源各找一个：
   - 头部开源项目（Star / 活跃度 / 维护方权威）
   - 学术论文或权威博客（设计理由）
   - 商业产品文档（真实场景下的取舍）
3. 快速过一遍，筛出**值得深读**的 2–3 个

产出：参考清单 + 每个参考的一句话定性。

---

## Phase 2 — Synthesize

目标：抽取**共性模式**与**关键差异**。

方法：画一张对比表：

| 维度 | Ref A | Ref B | Ref C |
|---|---|---|---|
| 核心抽象 | ... | ... | ... |
| 数据模型 | ... | ... | ... |
| 一致性保证 | ... | ... | ... |
| 故障模式 | ... | ... | ... |
| 性能 trade-off | ... | ... | ... |
| 运维成本 | ... | ... | ... |

关键问题：
- 哪些维度**所有参考都一致**？→ 这是行业共识，默认采纳
- 哪些维度**差异大**？→ 这是需要结合本项目约束做选择的点

产出：共性清单 + 差异表 + 自己的判断。

---

## Phase 3 — Adapt

目标：把共性 + 判断迁移到 Morrow 的约束下。

步骤：
1. 列出 Morrow 的约束：技术栈 / 团队规模 / 性能目标 / 合规 / 现有代码
2. 针对每个"差异点"，说明本项目倾向哪个方向、为什么
3. 形成**本项目的方案草案**（可能是多个参考的混合）

禁止：直接贴一个参考实现的设计就称"我们也这么做"。

产出：方案草案 + 每个决策点的依据。

---

## Phase 4 — Verify

目标：在投入大规模实现前，验证关键假设。

方法：
- 选出方案中 **最不确定 / 最高风险** 的 1–2 个假设
- 用最小实验验证（原型、benchmark、PoC）
- 验证失败 → 回 Phase 2 或 Phase 3 修订

产出：验证结果 + 修正后的方案 → 可以进入正式 Spec。

---

## 与 Spec 的接口

Research 的产出**直接进入 Spec 的"架构与技术思路"章节**：

- 参考清单 + 对比表 → Spec 的 Context / Alternatives
- 方案草案 + 判断依据 → Spec 的 Decision / 实现要点
- Verify 结果 → Spec 的 Feasibility 结论

---

## 反模式

- ❌ 找一个看起来最大的开源项目，直接模仿
- ❌ 只看"它怎么做"，不问"它为什么这么做"
- ❌ 跳过 Adapt，忽视本项目约束
- ❌ 跳过 Verify，把未验证的假设当既成事实写进 Spec
