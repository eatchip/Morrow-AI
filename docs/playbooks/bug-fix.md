# Playbook: Bug 修复

> 核心信条：**每个真实 bug 必须留下可复用资产**（测试 / 断言 / 规则 / 文档）。

---

## Step 1 — 复现

- 在最接近用户环境的层面复现（而不是单元测试臆想）
- 最小化复现步骤（去掉无关依赖，缩到最短路径）
- 如果复现不稳定：**先解决稳定复现再修**，飘忽 bug 的"修复"常是假象

复现失败就停下：
- 信息不足 → 回用户要：版本 / 环境 / 日志 / 操作序列
- 不是 bug 而是预期行为 → 改用文档 / UX 解决

---

## Step 2 — 定位

- 先形成 **2–3 个假设**，不要定点跳
- 用日志 / debugger / 二分 commit 验证假设
- 找到根因而非症状：能说出"为什么这个代码会产生这个结果"

**禁止**：症状消失就收工。没找到根因的"修复"要么假修，要么下次换形式复现。

---

## Step 3 — 测试先行

- 写一个**能稳定重现 bug** 的测试（Red）
- 测试应放在最合适的层级：
  - 逻辑错误 → unit
  - 跨模块行为 → contract / integration
  - 用户感知路径 → e2e
- 确认这个测试**在未修复代码下失败、修复后通过**

---

## Step 4 — 最小修复

- 只改**引入 bug** 的那层
- 不顺手重构（重构另起任务）
- 不添加未被需要的防御性代码

如果修复一发牵动多个模块，**升级为 Large 任务**，回到 `new-feature.md` 的 Spec → Plan 流程。

---

## Step 5 — 验证

- 修复后的测试通过（Green）
- 周边相关测试全绿
- 手工走一遍原始复现路径

---

## Step 6 — 留下资产

至少选一项：

- [ ] 新测试覆盖（Step 3 产物）
- [ ] 新增断言 / 不变量检查
- [ ] 更新 `DEVELOPMENT.md` / playbook 中的风险清单
- [ ] 起一条 ADR（如果 bug 揭示了架构缺陷）
- [ ] 在 `CHANGELOG.md [Unreleased] · Fixed` 登记

---

## Step 7 — Handoff

PR 描述必须包含：

- 根因分析（为什么会错）
- 修复方式（为什么这样改）
- 覆盖测试（如何防止再出现）
- 影响范围（是否可能有相似 bug 在别处）

---

## 反模式（不要做）

- ❌ 没有复现就开始改代码
- ❌ 症状消失就收工，不挖根因
- ❌ 修 bug 顺手重构无关代码
- ❌ 没写测试就 merge
- ❌ "这次先改着下次再加测试"
