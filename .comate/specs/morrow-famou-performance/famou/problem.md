## 1. Task Definition

- Core problem description:
  - Optimize Morrow Channel Workspace derived-view computation for long histories and many roles.
  - A candidate solution is a Python module that implements a deterministic projection strategy for channel data.
  - The projection must preserve the exact semantic output of the baseline implementation while improving latency and allocation proxy under large synthetic workloads.
- Input:
  - Synthetic `ChannelSnapshot`-like JSON data generated by the evaluator.
  - An `active_channel_id`.
  - Optional incremental update operations generated by the evaluator in later rounds.
- Output:
  - A derived view model containing:
    - `active_channel`
    - `channel_roles`
    - `ordered_events`
    - per-event role/run/handoff lookup data
    - `available_roles`
    - optional cache/update metadata if the candidate uses an incremental strategy
- Primary optimization objective:
  - Maximize correctness-preserving speedup over the baseline projection.
  - Correctness is a hard gate. Any semantic mismatch receives score `0`.
- Key metrics and formulas:
  - `correctness`: exact equality with the evaluator baseline oracle across all workloads.
  - `latency_ms_p95`: p95 elapsed time over repeated projection calls.
  - `speedup`: `baseline_latency_ms_p95 / candidate_latency_ms_p95`.
  - `memory_proxy`: peak size of candidate-owned indexes/caches measured by recursive object sizing.
  - `complexity_penalty`: small penalty for excessive candidate code size or unsafe constructs.
  - Proposed score:
    - `0` if correctness fails.
    - Otherwise `100 * log2(speedup + 1) - memory_penalty - complexity_penalty`.

## 2. Data Description

- Data sources:
  - The evaluator generates deterministic synthetic datasets based on Morrow channel concepts:
    - channels: `id`, `name`, `memberRoleIds`, timestamps
    - roles: `id`, `name`, `defaultRuntime`, timestamps
    - events: `id`, `channelId`, `type`, optional `roleId`, optional `runId`, optional `handoffId`, optional `text`, `createdAt`
    - runs: `id`, `channelId`, `roleId`, `status`
    - handoffs: `id`, `channelId`, `fromRoleId`, `toRoleId`, `status`
- Access method:
  - Generated in-memory by `evaluator.py`; no real user data and no network access.
  - Workload sizes should include small, medium, and large scales.
- Data quality and preprocessing assumptions:
  - IDs are strings and unique within their collection.
  - Events may arrive unsorted; output events must be ordered by `createdAt` ascending, with stable tie-breaking by original input order.
  - Some event references may point to missing roles/runs/handoffs; candidate behavior must match baseline behavior.

## 3. Constraints and Evaluation Basis

- Hard constraints:
  - Deterministic output for the same input.
  - No file system writes.
  - No network calls.
  - No subprocess creation.
  - No mutation of evaluator-owned input objects.
  - Exact semantic equivalence with the baseline oracle.
- Soft constraints / optimization targets:
  - Lower p95 projection latency.
  - Lower candidate-owned memory proxy.
  - Better behavior as event count grows from hundreds to tens of thousands.
  - Prefer strategies that can be ported cleanly to TypeScript/React without new top-level dependencies.
- How solution quality should be measured:
  - The evaluator runs correctness cases first.
  - If correct, it runs repeated timing cases with fixed random seeds.
  - The final score combines latency speedup, memory proxy, and complexity penalty.

## 4. Initial Solution Direction

- Baseline direction:
  - Filter active channel events from the full event list.
  - Sort by `createdAt`.
  - For each event, linearly search roles/runs/handoffs as needed.
  - Compute channel roles by mapping `memberRoleIds` through the roles list.
  - Compute available roles by filtering roles not in `memberRoleIds`.
- Required properties of a valid initial solution:
  - Simple, readable, and semantically identical to current Morrow renderer behavior.
  - No caching required for the initial solution.
  - Acts as a correctness reference and a low-performance baseline for FaMou to improve.
